环境

  • 游戏：FINAL FANTASY VII REMAKE Intergrade (Steam)
  • 引擎：Unreal Engine 4.18（Square Enix 定制版，Clang 编译）
  • 游戏路径：E:\SteamLibrary\steamapps\common\FINAL FANTASY VII REMAKE\
  • 游戏可执行文件：End\Binaries\Win64\ff7remake_.exe

一、UE4SS 安装与修复

1.1 安装

将 UE4SS v3.0.1 文件复制到游戏 Win64 目录：

UE4SS.dll
dwmapi.dll
UE4SS-settings.ini
Mods/  (整个文件夹)
1.2 遇到的第一个问题：RenderDoc 冲突

游戏目录下同时存在 dwmapi.dll（UE4SS）和 dxgi.dll（RenderDoc v1.44）， 两者都是代理 DLL，同时加载导致游戏无法启动。

解决：将 dxgi.dll 重命名为 dxgi.dll.bak。

1.3 遇到的第二个问题：GUObjectArray 扫描失败

UE4SS 日志显示：

[PS] Found EngineVersion: 4.18
[PS] Failed to find GUObjectArray: expected at least one value
[PS] Scan failed

原因：Square Enix 使用 Clang 编译 FF7R，生成的机器码与 UE4SS 内置的 MSVC 签名不匹配。 GUObjectArray 是所有功能的核心数据结构，找不到它就无法使用任何功能。

解决：从 UE4SS 官方 experimental-latest 的 zCustomGameConfigs.zip 中提取 FF7 Remake 专用配置：

文件 作用
UE4SS_Signatures/GUObjectArray.lua Clang 编译适配的 AOB 签名
UE4SS-settings.ini FF7R 专用参数（引擎 4.18，禁用 UObjectArrayCache）
MemberVariableLayout.ini Clang 编译器下的类成员偏移
VTableLayout.ini Clang 编译器下的虚表布局

关键设置：

[EngineVersionOverride]
MajorVersion = 4
MinorVersion = 18

[General]
bUseUObjectArrayCache = false    # 防止启动崩溃

[Debug]
ConsoleEnabled = 1               # 使用外部控制台
GuiConsoleEnabled = 0            # 禁用 GUI（避免渲染冲突）

GUObjectArray.lua 的官方签名：

function Register()
    return "48 8B ?? ?? ?? ?? ?? 4C 8B 04 C8 4D 85 C0 74 07 ?? ?? ?? 0F 94"
end

function OnMatchFound(matchAddress)
    local nextInstr = matchAddress + 0x7
    local offset = matchAddress + 0x3
    local dataMoved = nextInstr + DerefToInt32(offset) - 0x10
    return dataMoved
end

二、生成 .usmap 映射文件

2.1 快捷键配置

小键盘不可用，将快捷键改为字母键。编辑 Mods/Keybinds/Scripts/main.lua：

["DumpUSMAP"] = {["Key"] = Key.M, ["ModifierKeys"] = {ModifierKey.CONTROL}},
2.2 生成
  1. 启动游戏到主菜单
  2. 按 Ctrl + M
  3. 生成文件：Mappings.usmap（1.2 MB）

三、FModel 资产导出（失败）

3.1 配置
  • UE Version: 4.18
  • AES Key: 0x23989837645C9D28BA58072B2076E895B853A7C9E1C5591B814C4FD2A2D7B782
  • Mapping File: Mappings.usmap
3.2 现象

能浏览资产列表但保存失败：

[ERR] Could not save 'PC0000_00_BodyA_B'

原因：FModel 对 FF7R 定制引擎的骨骼网格体导出支持不完整。

四、UModel 资产导出（成功）

4.1 工具

需要使用 FF7R Intergrade 专用版 UModel（标准版无法处理 Square Enix 的定制 LZX 压缩）。

文件：umodel_FFVII_intergrade.zip（内含 v6 和 v7_test）。

4.2 启动命令
umodel_FFVII_intergrade_v7_test.exe -game=ue4.18 -path="E:\SteamLibrary\steamapps\common\FINAL FANTASY VII REMAKE\End\Content\Paks" -aes=0x23989837645C9D28BA58072B2076E895B853A7C9E1C5591B814C4FD2A2D7B782 -gui
4.3 使用
  1. 运行 Launch.bat
  2. 按 O 打开资源浏览器
  3. 导航到目标模型（如 Character/Player/PC0000_00_Cloud_Standard/Model）
  4. 选择模型文件，Ctrl+X 导出，选择 PSK 或 glTF 格式

五、关键教训

问题 根因 解决思路
游戏启动崩溃 多个代理 DLL 冲突 检查 Win64 目录下的非游戏 DLL
UE4SS 不工作 Clang vs MSVC 签名不匹配 使用官方 CustomGameConfigs
FModel 无法保存 工具对定制引擎支持不足 换用专用版 UModel
UModel 解压失败 标准版不支持定制 LZX 使用 Intergrade 专用编译版
AES 密钥不确定 原版和 Intergrade 密钥相同 直接从官方密钥库获取

Chapter 02

探索 AI 逆向 Drawcall 到材质球的落地尝试

这次实验从 PIX 捕获和导出的 drawcall 逆向资料出发，让 AI 先阅读 GPU 事件、PSO、资源绑定和反汇编摘要，再把能稳定迁移的材质侧计算落到 Unreal 项目资产中。目标不是完整复刻引擎 GBuffer，而是先验证一条可重复的工作流：从捕获证据到材质图，再到项目内可观察资产。

2.1 输入资料
  • E:\PIX\Cloud_DrawCall_Reverse.md：drawcall 候选、资源绑定和材质族群摘要。
  • E:\PIX\Cloud_Draw171_PSO2019_Material_Repro.md：draw 171 / GlobalId 237 / PSO 2019 的落地复现说明。
  • E:\PIX\extracted_shaders\pso_2019_ps.asm：像素着色器反汇编。
  • E:\PIX\GPU_3_export_cpp：PIX replay 导出的 D3D12 C++ 现场。
2.2 关键判断

draw 171 是 BasePass-like 绘制：PSO 2019 写入多个 render target，深度测试为 Equal 且不写深度，符合 Unreal 风格的 GBuffer 填充。运行时 descriptor table 指向 512 贴图族：t4 为 albedo，t1/t2/t3 参与法线混合，t5/t6 提供 roughness/material scalar，t7 提供 alpha/mask。

2.3 落地到 Unreal 材质
PIX 槽位 逆向含义 Unreal 材质输入
t4 颜色 / albedo Base Color
t1 + t2 + t3 RG normal decode、detail/base normal blend Normal，使用 Custom HLSL 节点
t5 + t6 + t3 roughness seed 与重映射 Roughness，按反汇编公式近似
t7 alpha / mask Opacity Mask
2.4 项目内生成结果
  • 生成材质母版：/Game/Mesh/CloudTexture/PIXMaterials/M_Cloud_PIX_Draw171_Repro
  • 生成材质实例：/Game/Mesh/CloudTexture/PIXMaterials/MI_Cloud_PIX_Draw171_Eye_Repro
  • 写入网格槽位：/Game/Mesh/PC0000_00 的 PC0000_00_Eye slot 指向新实例。
  • 验证记录：Saved/MaterialReverse/cloud_pix_draw171_landing_verification.md
2.5 踩坑与修正
  1. UE 5.5 Python 绑定中没有 unreal.ExpressionInput，Custom 节点输入需要先创建 CustomInput，再通过 MaterialEditingLibrary.connect_material_expressions 按输入名连接。
  2. Custom HLSL 代码被内嵌进 UE 生成函数，不能在节点代码里再定义局部函数。原本的 DecodeRGNormal 函数导致 SM6 报错，最终改成纯内联表达式。
  3. 体积光照、GBuffer packing、RT 输出编码属于场景和渲染管线层，不适合直接塞进普通材质球。本次只落地材质侧可观察近似。
2.6 工作流结论

AI 在这类任务里的价值主要不是“猜材质”，而是快速对齐多源证据：PIX 事件列表、descriptor 最终写入、shader 反汇编、UE 材质 API、编译日志。最有效的节奏是先把 drawcall 拆成可验证的材质片段，再用脚本生成资产，最后用编辑器日志和验证脚本闭环。
